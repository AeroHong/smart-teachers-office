## 1. 한 줄 소개

**스마트교무실**은 선유고등학교 교사가 직접 기획하여 만든 "학교 맞춤형 경량 업무 시스템"입니다.
NEIS(나이스)가 법정 공식 기록을 위한 시스템이라 유연하지 못한 부분 — 매 시간 QR로 출석 체크하기, 결강 시 보강 교사 빠르게 구하기, 연수 참석 서명 받기 — 를 보완합니다. NEIS를 대체하지 않고, NEIS가 다루지 않는 "현장 실무의 빈틈"을 메우는 것이 목표입니다.

2026년 4월 첫 배포 이후 선유고에서 실사용 중이며, 약 3개월간 v1.0 → v3.0으로 계속 개선되어 왔습니다.

## 2. 왜 만들었나 — 해결하는 문제

| 기존 방식 | 문제점 | 스마트교무실 |
|---|---|---|
| 조회·수업 시간 구두 호명 출석 | 시간 소요, 기록 누락, 지각/외출 관리 어려움 | QR 스캔 1초 체크인, 실시간 현황판 |
| 결강 시 전화·메신저로 보강 교사 수소문 | 비효율, 기록 안 남음 | 온라인 신청·수락 게시판, 이력 자동 기록 |
| 연수·회의 참석 종이 서명부 | 관리·보관 번거로움 | QR 스캔 후 전자 서명, 명단 자동 집계 |
| 개인별 공지(수행평가 일정 등) 전달 | 문자·구두 전달, 확인 여부 불명 | 학생별 공지 등록 + 확인 여부 추적 |

## 3. 핵심 기능 (4개 모듈)

### 3-1. 스마트 출결
- 조회·수업·방과후·행사 등 이벤트별 QR 코드 생성, 학생은 스마트폰 스캔만으로 출석
- **라이브 세션**: 출석 시작 버튼 → 수업시간 1/3 경과 시 QR 자동 마감 → 종료 시 미출석 자동 결석 처리 (사람이 직접 마감하지 않아도 됨)
- 외출·복귀 기록, 수업시간 1/3 초과 시 경고 표시
- 결석 사유 프리셋(질병·조퇴·지각·체험학습 등) + 직접 입력
- 반복 일정(예: 매주 화·목 3교시) 자동 생성, 달력으로 과거 출결 조회
- 통계 대시보드 — 이벤트별/학생별 출결 현황, 결석 이력 조회
- 출석 1~2~3등 메달 표시, 전원 출석 시 축하 애니메이션(동기부여 요소)

### 3-2. 보강 신청
- 결강 정보(날짜·교시·과목·반)와 보강 방식(자습/과제/대체수업)을 온라인 등록
- 신청 가능·예정·마감 상태를 카드로 실시간 표시, 교사가 원클릭 신청
- 관리자가 직접 보강 교사 지정도 가능
- 내 신청 내역, 전체 현황판(월별 필터), Excel 내보내기
- 월별/전체 보강 지원 횟수 랭킹("명예의 전당")

### 3-3. 연수 서명부
- 연수명·날짜·장소·대상자 등록 → QR 스캔 후 전자 서명(터치/마우스)
- 서명 현황 실시간 집계, 자주 쓰는 참석자 명단은 프리셋으로 저장·재사용

### 3-4. 스마트 공지 + 학생 포털
- 교사가 이벤트(수업)별로 전체/개인 대상 공지 작성
- 학생은 자신의 구글 계정으로 로그인해 본인에게 해당하는 공지만 확인, 확인 여부가 교사에게 집계됨

## 4. 도입 시 기대 효과

- **시간 절감**: 매시간 호명 출석 대신 QR 스캔 → 수업 시작 지연 최소화
- **기록 정확도**: 수기 기록·구두 전달의 누락·오기 위험 감소, 결석·보강 이력이 검색 가능한 형태로 축적
- **실시간성**: 교사·관리자가 새로고침 없이 즉시 현황 확인 (Firestore 실시간 구독)
- **투명성**: 보강 신청·승인, 공지 확인 여부가 모두 이력으로 남아 사후 확인 가능

다만 이는 선유고 자체 운영 경험에 기반한 정성적 효과이며, 별도의 정량적 효과 측정(설문·시간 측정 등)은 아직 수행되지 않았습니다. 도입 검토 시 이 점을 감안해 주세요.

## 5. 보안 및 개인정보 처리 — 비개발자를 위한 설명

**Q. 우리 학교 자료를 다른 학교 선생님이 볼 수 있나요?**
아니요. 모든 데이터는 학교 단위로 데이터베이스 규칙 수준에서 분리되어 있습니다. 다만 "어떤 학교가 존재하는지"를 확인하는 최소한의 정보(학교 이름 등)는 로그인한 사용자라면 조회가 가능한 구조입니다 (신규 학교가 시스템에 가입할 때 학교를 검색하기 위한 기능). 학생 명단·출결 기록·보강 신청 등 실제 업무 데이터는 소속 학교 구성원만 접근 가능합니다.

**Q. 로그인은 어떻게 하나요? 비밀번호를 따로 관리하나요?**
학교 구글 계정(Google Workspace)으로만 로그인합니다. 비밀번호는 시스템에 저장되지 않고 구글이 인증을 전담하며, 구글 계정에 2단계 인증을 설정하면 그대로 적용됩니다.

**Q. 아무나 가입해서 우리 학교 자료에 접근할 수 있나요?**
가입 신청 자체는 누구나 할 수 있지만(구글 계정만 있으면), 학교 관리자가 승인하기 전까지는 어떤 데이터도 볼 수 없습니다("승인 대기" 상태). 관리자가 교직원 이메일을 미리 등록해두면(사전 등록) 해당 교사는 로그인 즉시 자동으로 승인됩니다.

**Q. 퇴직·전출 교사 계정은 자동으로 정리되나요?**
아니요, 이 부분은 수동입니다. 인사이동 시 학교 관리자가 관리자 페이지에서 직접 계정을 비활성화해야 합니다. 도입 시 이 절차를 인사 담당자와 공유하는 것을 권장합니다.

**Q. 개인정보처리방침이 있나요?**
네, 별도 페이지(`/privacy-policy`)로 공개되어 있으며 인쇄용 문서도 마련되어 있습니다. 안전성 확보조치 조항까지 포함해 최근 개정되었습니다.

## 6. 도입 요건

| 항목 | 필요 여부 | 비고 |
|---|---|---|
| 학교 Google Workspace (구글 계정) | **필수** | 로그인이 구글 OAuth 기반이라 학교 구글 계정 체계가 없으면 도입이 어렵습니다 |
| 학생 이메일 형식 | 권장 | 학번을 포함한 이메일 패턴(숫자 9자리 등)이 있으면 학생 로그인·자동 인증이 매끄럽습니다. 없어도 교사가 수동 등록 가능 |
| 학생 스마트폰 보유 | 사실상 필수 | QR 체크인 방식이라 학생 다수가 스마트폰으로 스캔 가능해야 합니다 |
| 별도 서버 구축 | **불필요** | Firebase 클라우드 기반, 학교에서 서버를 운영할 필요 없음 |

## 7. 도입 절차 (예상)

1. **학교 등록**: 시스템 내 "학교 설정" 화면에서 학교 검색 → 없으면 신규 등록 (최초 등록자가 학교 관리자가 됨)
2. **사전 등록(선택)**: 학교 관리자가 교직원 이메일 명단을 CSV로 일괄 업로드 → 해당 교사는 첫 로그인 시 승인 절차 없이 바로 진입
3. **교사 가입 승인**: 사전 등록이 안 된 교사는 로그인 후 "승인 대기" 상태가 되며, 관리자가 이메일을 확인하고 승인
4. **학생 명단 업로드**: 학년·반·번호·이름 등을 CSV로 일괄 등록 (선택과목 분반의 경우 별도 그룹 관리)
5. **운영 시작**: 출결 이벤트 생성, 보강 신청, 연수 등록 등 순차적으로 사용

현재는 선유고 1개교 실사용 + 문의 학교 대응 단계이며, 표준화된 "온보딩 매뉴얼"은 아직 초안 수준입니다. 도입 학교와 함께 다듬어가는 것을 전제로 진행하는 것이 현실적입니다.

## 8. 운영 비용 및 유지보수 구조

- 별도 서버·호스팅 비용 없이 Google Firebase 클라우드 인프라(종량제)를 사용합니다.
- Cloud Functions(자동 결석 처리 등) 사용을 위해 유료 요금제(Blaze) 전환이 필요하지만, 학교 1곳 규모의 트래픽에서는 대부분 무료 제공량 범위 내이거나 매우 낮은 수준의 실사용료만 발생하는 구조입니다. (실제 학교별 청구 금액은 트래픽에 따라 달라질 수 있어 정확한 수치는 별도 확인이 필요합니다.)
- 시스템 유지보수·기능 개선은 현재 선유고 교사 1인(개발자 본인)이 AI 코딩 도구(Claude Code)를 활용해 직접 수행하고 있습니다. 별도의 전담 개발팀이나 유지보수 계약이 있는 상용 서비스가 아니라는 점을 분명히 안내드립니다.

## 9. 현재 한계와 유의사항 (투명성을 위한 안내)

도입 결정에 참고하실 수 있도록, 실제 상태를 있는 그대로 안내드립니다.

- **1인 개발·운영 체제**: 현직 교사가 본업과 병행하여 개발·운영합니다. 상용 SaaS처럼 24시간 장애 대응이나 SLA(서비스 수준 보장)를 제공하지는 않습니다.
- **자동화된 테스트 체계 없음**: 별도의 자동 테스트 코드 없이, 기능 변경 후 실제 운영 환경에서 확인하며 배포하는 방식입니다. 대규모 동시 접속 등 극단적 상황에 대한 사전 부하 테스트는 수행되지 않았습니다.
- **접근 제어 세부 조정 진행 중**: 신규 학교가 시스템 가입 시 학교를 검색할 수 있도록, 로그인한 사용자는 다른 학교의 최소한의 식별 정보(학교명 등)를 조회할 수 있는 구조가 남아 있습니다. 학생 개인정보·출결기록 등 민감 데이터는 소속 학교 구성원만 접근 가능하도록 별도로 차단되어 있습니다.
- **계정 회수는 수동**: 퇴직·전출 교사 계정은 관리자가 직접 비활성화해야 하며, 자동 만료 기능은 없습니다.
- **다학교 운영 경험 아직 적음**: 현재까지 실제 서비스 대상은 선유고 1개교이며, 여러 학교를 동시 지원한 운영 경험은 누적 중입니다.
- **일부 기능은 계획 단계**: 공지 이메일 자동 발송, 교사 시간표 자동 입력 연동 등은 아직 구현되지 않은 로드맵 항목입니다.

## 10. 향후 개발 방향 (로드맵)

| 단계 | 내용 |
|---|---|
| 진행 중 | 학생 포털 고도화, 스마트 공지 확인 추적 |
| 계획 | 공지 이메일 자동 발송(교사 이름으로 발신) |
| 중기 | 교사 시간표 입력 → 출결 이벤트 자동 생성 |
| 장기 | 학교 자체 수강신청 데이터와 연동해 학생-분반 명단 자동 매핑 |
| 장기 | 다학교 확장(여러 학교가 하나의 시스템을 각자 독립적으로 사용) — 현재도 구조적으로 지원되며 계속 보완 중 |

## 11. 도입 적합성 자가 체크리스트

아래 항목에 다수 해당할수록 도입 시 시행착오가 적습니다.

- [ ] 학교가 Google Workspace(구글 계정) 체계를 이미 사용하고 있다
- [ ] 학생 대부분이 QR 스캔이 가능한 스마트폰을 소지하고 있다
- [ ] 시스템을 처음 세팅하고 안내할 학교 관리자(또는 담당 교사)를 지정할 수 있다
- [ ] "상용 서비스 수준의 SLA는 없는 현직 교사 1인 운영 시스템"이라는 점을 이해하고 수용할 수 있다
- [ ] 초기 도입 과정에서 일부 매뉴얼·절차를 함께 다듬어갈 여지가 있다
