# 단위학교 과목별 성취평가 운영 점검 체크리스트 - 웹서비스 개발 스펙

> Claude Code 작업 인계용 문서. Claude.ai 채팅에서 요구사항 정리 및 원본 체크리스트 파싱까지 마친 상태.
> 기존에 운영 중인 `smart-teachers-office`(멀티테넌트 SaaS, Firebase Auth + role-based auth)와
> 동일한 스택/패턴을 재사용하는 것을 전제로 설계함.

---

## 1. 배경 / 목적

교육청 공문으로 "단위학교(과목별) 성취평가 운영 점검 체크리스트" 2종이 내려옴.
- 각 과목마다 여러 명(1~5명)의 교사가 공동으로 체크리스트를 작성하고 서명해야 함
- 관리자(교무부장 등)가 과목별로 취합해서 PDF로 제출해야 함
- 현재는 종이/한글파일로 개별 작성 → 취합이 비효율적 → 웹서비스로 대체

**주의(범위 외 이슈)**: 작성된 체크리스트 최종 제출처는 "성취평가 아카이브" 사이트인데,
2학년 5단계 평가 과목이 학년 선택 시 안 뜨고 직접입력 드롭다운에도 없는
아카이브 시스템 자체의 데이터 매핑 오류가 있음. 이건 교육청 담당자에게 별도 문의 중이며
본 웹서비스 개발 범위와는 무관 (내부 취합용 도구이므로 아카이브 문제와 독립적으로 진행).

## 2. 원본 문서 2종 구조

### 2-1. 붙임1: 단위학교(과목별) 성취평가 운영 "과정" 점검 체크리스트

**기본 정보 필드**
- 학교명
- 점검 과목
- 공동 출제 교사 (서명란 최대 3~5명, 과목마다 인원 다름 → 동적으로 처리 필요)

**체크리스트 문항** (총 19문항, 5개 단계로 그룹핑)

각 문항은 `예/아니오` 선택 + `점검내용 준수 여부 파악 시 확인한 근거자료 체크박스`(복수선택) 구조.

### 2-2. 붙임2: 단위학교(과목별) 성취평가 운영 "결과" 점검 체크리스트

**기본 정보 필드**: 붙임1과 동일 (학교명, 점검 과목, 공동 출제 교사 서명 여러 명, 점검일자, 확인자(교감) 서명)

**1단계: 조건부 게이트 문항 (2문항, 예/아니오)**

| 번호 | 문항 |
|---|---|
| 1 | 성취수준 A의 분할점수가 과목 평균보다 낮게 나타났는가? |
| 2 | 성취수준 A의 비율이 23.9%*보다 높게 나타났는가? (*2025년 1,2학기 성취도 5단계 적용 과목의 성취수준 A의 전국 상위 30% 값) |

> **분기 로직**: 위 1, 2번 중 하나라도 '예'가 있으면 → 아래 2단계 체크리스트 작성 + 종합분석 의견 필수. 둘 다 '아니오'면 2단계는 건너뛸 수 있음(UI에서 조건부로 노출/숨김 처리 필요).

**2단계: 단계별 성취평가 결과 점검 (12문항, 5개 카테고리, 예/아니오 + 비고(특이사항) 서술란)**

| 단계 | 번호 | 문항 |
|---|---|---|
| 평가계획 및 평가문항 출제 | 1 | 과목별 성취기준에 따른 성취수준(또는 평가기준)을 마련하여 이에 근거하여 평가 계획을 수립하였는가? |
| 평가계획 및 평가문항 출제 | 2 | 성취수준별 추정분할점수 산출 방식을 따를 경우 교과 특성, 평가문항의 난이도 등을 고려하였으며 담당 교사의 협의 과정을 거쳤는가? |
| 평가계획 및 평가문항 출제 | 3 | 1차 고사의 분석 결과(문항 난이도, 변별도, 정답률, 오답률 등)을 2차 고사의 출제시 반영하였는가? |
| 평가계획 및 평가문항 출제 | 4 | 동일 교과(목) 담당 교사 간 공동 출제를 하고, 교과 내 검토를 강화하였는가? |
| 평가계획 및 평가문항 출제 | 5 | 평가 출제시 각 성취수준 도달 여부를 판단할 수 있는 다양한 수준의 문항을 출제하였는가? |
| 분할점수 산출 | 6 | 성취수준별 추정분할점수 적용 시, 나이스 분할점수 기능 또는 분할점수 산출 프로그램을 사용하였는가? |
| 분할점수 산출 | 7 | 성취수준별 추정분할점수 적용 시, 공동 출제 교사들이 모두 참여하였는가? |
| 평가시행 및 결과 분석 | 8 | 채점에서 타당성과 객관성을 확보하는가? (채점자 간 일치도, 특정 집단에 유리하지 않은 기술) |
| 평가시행 및 결과 분석 | 9 | 성취수준별 비율이 특정 성취수준에 편중된 경우 교과협의회 등을 통해 원인을 분석하였는가? |
| 평가시행 및 결과 분석 | 10 | 평가문항을 분석함으로써 출제한 문항이 다양한 성취수준에 부합한지 판단하고 성찰하였는가? |
| 평가시행 및 결과 분석 | 11 | 성취수준 판별과 무관한, 지나치게 쉬운 문항이나 어려운 문항을 출제하지는 않았는가? |
| 평가시행 및 결과 분석 | 12 | 학생들의 학업 성취 정도를 고려하였을 때, 우리 학교의 성취수준 A의 비율은 적절한 것으로 판단할 수 있는가? |

**종합분석 의견**: 서술형 텍스트 (장문, textarea)

## 3. 요구사항 확정 사항 (사용자 답변)

- 체크리스트 구조: 체크 + 서술형 혼합 (위 표 참고)
- 서명 방식: **마우스/터치로 직접 그리기** (canvas 기반 signature pad)
- 로그인: 학교 Google Workspace 계정, 사용자는 관리자가 미리 등록
- 대상 과목: 과목별로 지정되어 있고, 과목당 참여 교사 1~5명으로 다양함
- 관리자: 과목별로 작성된 체크리스트를 PDF로 다운로드하여 제출

## 4. 아키텍처 제안

### 4-1. 인증
- Firebase Auth + Google Identity Provider, hosted domain 제한(`@학교도메인`)으로 학교 계정만 로그인 허용
- role 구분: `admin`(교무부장 등) / `teacher` — 기존 `smart-teachers-office`의 role-based auth 패턴 재사용
- 로그인 이메일이 사전 등록 명단(`teachers` 컬렉션)에 없으면 접근 차단

### 4-2. 사용자(교사) 사전 등록
- 관리자가 엑셀/명단 업로드 → Firestore `teachers` 컬렉션 (이메일, 이름, 담당 과목 매핑)
- 과목(`subjects`) ↔ 교사(`teachers`) N:M 매핑 필요 (한 교사가 여러 과목, 한 과목에 여러 교사)

### 4-3. 데이터 모델 (Firestore, 초안)

```
subjects/{subjectId}
  - name: string (과목명)
  - grade: number (학년)
  - assignedTeacherUids: string[] (해당 과목 공동출제 교사 uid 목록, 1~5명)
  - homeroomTeacherUid?: string (필요시)

checklistSubmissions/{submissionId}
  - subjectId: string
  - checklistType: "process" | "result"  // 붙임1 / 붙임2
  - schoolName: string
  - checkDate: timestamp
  - status: "draft" | "in_progress" | "submitted" | "locked"
  - answers: {
      [questionId]: {
        value: "예" | "아니오",
        evidenceChecks?: string[],   // 붙임1 근거자료 체크박스
        note?: string                 // 비고/특이사항
      }
    }
  - comprehensiveOpinion?: string     // 붙임2 종합분석 의견
  - signatures: {
      [teacherUid]: {
        signatureImageUrl: string,    // Storage 경로
        signedAt: timestamp,
        teacherName: string
      }
    }
  - confirmerSignature?: {            // 확인자(교감)
      signatureImageUrl: string,
      signedAt: timestamp
    }
  - createdAt / updatedAt: timestamp

teachers/{uid}
  - email: string
  - name: string
  - role: "admin" | "teacher"
  - subjectIds: string[]
```

> 붙임2는 1단계 게이트 문항(1,2번) 결과에 따라 2단계 12문항 + 종합분석의견 노출 여부가 갈리므로,
> `answers`에 게이트 문항 값도 포함하고 프론트에서 조건부 렌더링 처리.

### 4-4. 작성 + 서명 플로우
- 교사 로그인 → 본인이 배정된 과목 목록 확인 → 체크리스트 선택(과정/결과) → 항목 작성
- 공동 작성이므로 여러 교사가 동일 `submission` 문서에 각자 접속해서 항목 공동 편집 가능해야 함
  (동시 편집 충돌 최소화를 위해 문항 단위 업데이트 권장, 또는 "제출 확정" 전까지는 마지막 저장이 우선)
- 서명은 canvas 기반 signature pad(예: `signature_pad` 라이브러리)로 그린 후 PNG로 Storage 업로드,
  URL을 `signatures.{uid}`에 저장
- 모든 필수 서명(공동출제 교사 전원 + 확인자)이 완료되어야 `status: "submitted"`로 전환, 이후 수정 잠금

### 4-5. 관리자 대시보드 + PDF 다운로드
- 과목별 필터 / 상태별 필터(작성중/제출완료) 리스트
- 개별 PDF 다운로드 + 전체 일괄 zip 다운로드
- PDF 생성 방식 두 가지 옵션:
  - **클라이언트**: `jsPDF` + `html2canvas` — 서명 이미지 포함이 간단, 구현 빠름
  - **서버(Functions)**: Puppeteer로 HTML→PDF — 레이아웃 안정적, 한글 폰트 처리 확실
  - 원본 문서(붙임1/2) 서식과 최대한 유사하게 표 레이아웃 재현 권장 (제출용이므로)
